为了在 **4分钟内** 完成高质量视频生成，目前的**串行（Sequential）架构**是最大的瓶颈。

目前的流程是：
`剧本生成 -> (等待) -> 图片生成 -> (等待) -> 视频生成 -> (等待) -> 动画代码撰写 -> 最终HTML`

如果生成3个视频片段（假设每个1分钟）和5张图，再加上大模型思考，时间轻松突破5-8分钟。

**核心解法：将“资产生成（图片/视频）”与“代码构建（HTML/GSAP）”并行化（Parallelism），并采用“占位符+后期注入”的模式。**

以下是为您重构的 **Dify Workflow 高性能架构方案**。

---

### 1. 新的架构设计 (The Async-Parallel Architecture)

我们将利用 Dify Workflow 的 **并行分支 (Parallel Branch)** 功能。

**流程图：**

```mermaid
graph TD
    A[用户输入 Query] --> B(调度智能体 / 意图识别);
    B --> C{任务类型};
    
    C -- 修改/微调 --> D[调用修改Agent];
    C -- 初次生成 --> E[剧本生成 Agent];
    
    %% 关键优化点 %%
    E -- 输出 JSON 草稿 (含 Prompt) --> F(并行网关 Parallel);
    
    subgraph 并行处理阶段
        F --> G[分支1: 资产生产工厂 (Code Node)];
        G --> G1[并发调用图像生成 API];
        G --> G2[并发调用 Veo 视频生成 API];
        G1 & G2 --> G3[输出资产映射表 Asset_Map];
        
        F --> H[分支2: 动画构建 Agent];
        H --> H1[使用占位符编写 HTML/GSAP];
        H --> H2[输出 HTML 模板];
    end
    
    G3 & H2 --> I[合并节点 (Code Node)];
    I --> J[字符串替换: 将 HTML 中的占位符替换为真实 URL];
    J --> K[最终 HTML产物];
    K --> L[回复用户];
```

### 2. 详细调整方案

#### 第一步：改造“剧本生成智能体” (只做脑力活)

**修改点**：
1.  **移除所有工具调用** (`generate_nanobanana_images`, `veo_generator`)。
2.  **移除第二阶段（资产注入）**。
3.  **修改输出**：仅输出包含 Prompt 的 JSON。图片/视频字段不再是 URL，而是保留 `prompt` 和生成的 `uuid`。

**Prompt 修改建议 (截取核心变更部分)：**

```markdown
# 移除工具定义部分，增加 ID 生成规则

...
# 思考&执行工作流 (简化版)

1. 解析输入...
2. 编导规划...
3. 生成剧本草稿 (Generate Draft Script):
   * 基于规划生成 JSON。
   * 对于图像组件: 
     - "type": "ai-generation-image"
     - "id": "img_uuid_1234" (必须生成唯一ID)
     - "value": "详细的绘画提示词..."
   * 对于视频组件:
     - "type": "generative-video"
     - "id": "vid_uuid_5678" (必须生成唯一ID)
     - "content": { "prompt": "详细的视频生成提示词..." }
   * **注意**：你不需要调用工具去生成它们，只需要写好 Prompt 和 ID。

# 输出格式
(保持原 JSON 结构，但 value 字段里填的是 prompt)
```

#### 第二步：分支 1 - 资产生产工厂 (Python Code Node)

这是性能提升的关键。使用 Python 的 `concurrent.futures` 实现**并发请求**。即使 Veo 生成需要 60秒，如果我们同时发起 3 个请求，总耗时依然是 ~60秒，而不是 180秒。

**Dify 代码节点逻辑 (Python):**

```python
import json
import concurrent.futures
import requests

def main(script_json_str: str):
    """
    输入: 剧本 Agent 输出的 JSON 字符串
    输出: 一个字典 map, key 是 uuid, value 是生成的 URL
    """
    try:
        script_data = json.loads(script_json_str)
    except:
        return {"error": "Invalid JSON"}
        
    assets_map = {}
    tasks = []
    
    # 提取所有需要生成的任务
    # 递归遍历 JSON 找 type 为 ai-generation-image 或 generative-video 的节点
    # 这里简化逻辑，假设你通过遍历拿到了一个列表
    # item 结构: {'id': '...', 'type': '...', 'prompt': '...'}
    generation_tasks = extract_tasks_from_json(script_data) 

    def call_image_api(task):
        # 调用你的 nanobanana 或其他 API
        # url = request_image(task['prompt'])
        return task['id'], "https://mock_url/image.png" 

    def call_video_api(task):
        # 调用 Veo API
        # url = request_veo(task['prompt'])
        return task['id'], "https://mock_url/video.mp4"

    # 并发执行
    with concurrent.futures.ThreadPoolExecutor(max_workers=10) as executor:
        future_to_task = {}
        for task in generation_tasks:
            if task['type'] == 'ai-generation-image':
                future = executor.submit(call_image_api, task)
            elif task['type'] == 'generative-video':
                future = executor.submit(call_video_api, task)
            future_to_task[future] = task
            
        for future in concurrent.futures.as_completed(future_to_task):
            asset_id, asset_url = future.result()
            assets_map[asset_id] = asset_url
            
    return {
        "assets_map": assets_map,
        "original_script": script_data # 透传下去
    }

def extract_tasks_from_json(data):
    # 需要你自己实现递归查找逻辑
    tasks = []
    if isinstance(data, dict):
        if data.get('type') == 'ai-generation-image':
            tasks.append({'id': data.get('id'), 'type': 'ai-generation-image', 'prompt': data.get('value')})
        elif data.get('type') == 'generative-video':
             tasks.append({'id': data.get('id'), 'type': 'generative-video', 'prompt': data.get('content', {}).get('prompt')})
        
        for key, value in data.items():
            tasks.extend(extract_tasks_from_json(value))
    elif isinstance(data, list):
        for item in data:
            tasks.extend(extract_tasks_from_json(item))
    return tasks
```

#### 第三步：分支 2 - 动画生成智能体 (并行工作)

这个 Agent 不再等待图片生成好。它拿着 Prompt 和 ID 直接开始写代码。

**Prompt 修改建议：**

```markdown
# 角色：精英级动效图形工程师
...
## 0. 导演级剧本解读指南

### 关于媒体资源的特殊处理 (Critical Change)
你收到的 JSON 剧本中，图像和视频字段尚未生成 URL，而是包含了 `id` 和 `prompt`。
**你的任务是使用占位符 ID 编写代码。**

*   **图片处理**：
    当遇到 `type: "ai-generation-image"` 时，在 HTML/JS 中，将其 URL 暂时写为 `{{ASSET_ID}}` (例如: `{{img_uuid_1234}}`)。
    *不要使用假的 http 地址，直接使用双大括号包裹 ID。*

*   **视频处理**：
    当遇到 `type: "generative-video"` 时，同样将其 src 写为 `{{vid_uuid_5678}}`。

...
(其余关于 GSAP、布局、动画的规则保持不变)
...
```

#### 第四步：合并节点 (Assembly Node)

这是最后一步，将 Branch 1 的 Map 和 Branch 2 的 Template 结合。

**Dify 代码节点逻辑 (Python):**

```python
def main(html_template: str, assets_map: dict):
    final_html = html_template
    
    # 简单的字符串替换
    for asset_id, asset_url in assets_map.items():
        placeholder = "{{" + asset_id + "}}"
        final_html = final_html.replace(placeholder, asset_url)
        
    # 兜底：如果有些图生成失败了，替换成默认透明图或占位图，防止前端报错
    # final_html = regex_replace(r"{{.*?}}", "default.png", final_html)
    
    return {
        "result_html": final_html
    }
```

### 3. 为什么这个方案能达到 4 分钟？

我们算一笔账：

1.  **剧本生成 (LLM)**: ~45秒 (只出 JSON，不调工具)。
2.  **并行阶段 (Parallel Phase)**: 取两者中耗时较长者。
    *   *分支 A (资产)*:
        *   图片并发: ~20秒
        *   Veo 视频并发: 假设 90秒 (这是硬成本，但现在只算一次最慢的，不算累计)。
        *   **耗时: ~90秒**
    *   *分支 B (写代码)*:
        *   Claude 3.5 Sonnet / GPT-4o 写 300行代码: **~60-90秒**。
3.  **合并与输出**: ~5秒。

**总耗时预测**: 45s (剧本) + 90s (并行最大值) + 5s = **约 2分20秒**。
即使 Veo 偶尔慢一点，或者代码写得慢一点，你也有大量的 Buffer 保持在 4分钟以内。

### 4. 额外的优化建议

为了进一步提升用户体验和成功率：

1.  **预加载与 Loading 态**:
    *   在动画 Agent 的 Prompt 中，要求它编写一个 `Preloader`。
    *   即使视频没加载完，先显示 HTML 骨架和 Loading 动画，提升用户的“体感速度”。

2.  **兜底图片策略**:
    *   如果 Veo 失败了，不要让流程挂掉。在 Asset Factory 代码里做 `try...catch`，如果视频生成失败，返回一个静态图片的 URL（可以用之前的 Image Gen 模型生成一张做替补）。

3.  **HTML 结构缩减**:
    *   目前的 Prompt 非常长，包含大量教学内容。可以尝试将其精简，或者利用 Dify 的 **Few-Shot (少样本)** 功能，给 Agent 投喂 1-2 个完美的 HTML/JSON 对示例，减少它推理逻辑的时间，直接让它“照猫画虎”，这样生成代码的速度会快 30% 以上。

4.  **Token 优化**:
    *   剧本生成的 `tech_implementation_hint` 字段非常详细。如果下游的动画 Agent 足够聪明，可以将上游简化，只给风格关键词，让下游自己发挥，减少中间传输的 Token 量。

现在的瓶颈不在模型思考慢，而在于**串行 IO 阻塞**。

请立即将你的 Dify 应用从简单的 **Agent 模式** 切换为 **Workflow (工作流) 模式**，并按照上述的 **Parallel (并行)** 结构重组工具调用。这是解决速度问题的唯一架构级方案。
